# ドキュメント管理方針とテスト基本方針

## Docs管理方針（エージェント用含む）

### エージェント運用ルール（必須）

- すべて日本語で対応すること
- 作業前にプロジェクトコンテキストとドキュメント規約を必ず読み込む
- ドキュメントを作成/変更したらドキュメント一覧（INDEX）を更新する
- 実装指示を完了したら、関連ドキュメントの更新要否を必ず確認する
- 実装指示を完了したら、関連テストの作成/更新要否を必ず確認する
- ドキュメントのファイル名は英語、本文は日本語
- 自動コミット禁止（明示的に指示された場合のみ実行）

### 実装完了時の docs / test 確認ルール

- 実装や仕様変更を行った場合、完了報告前に「関連docsの更新が必要か」を確認する。
- 変更が必要な場合は、実装差分と整合するように docs を更新する。
- 変更が不要な場合も、不要と判断した理由（既存docsで十分、仕様差分なし等）を簡潔に記録・共有する。
- 実装や仕様変更を行った場合、完了報告前に「関連テストの作成/更新が必要か」を確認する。
- 変更が必要な場合は、実装差分と整合するように Unit / Component / E2E のいずれかでテストを追加・更新する。
- 変更が不要な場合も、不要と判断した理由（既存テストで十分、対象が設定のみ等）を簡潔に記録・共有する。

### ドキュメント命名規則

- ディレクトリ名: 英語の kebab-case
- ファイル名: 英語の kebab-case
- ファイル内容: 日本語

### INDEX更新ルール

- docs配下で新規作成・改名したら必ず更新
- フォーマット:

| File Path | Title / Description (Japanese) | Tags |
| :--- | :--- | :--- |

### Frontmatter（Markdown先頭に必須）

```yaml
---
title: 日本語のドキュメントタイトル
description: このドキュメントの概要
tags: [タグ1, タグ2]
---
```

## テスト基本方針

### テストの責務

- Unit（Vitest）: ロジック検証、カバレッジ主担当
- Component（React Testing Library）: コンポーネントの振る舞い検証
- E2E（Playwright）: ユーザーフローの結合検証

### Supabase テスト環境

- ローカル Supabase（`supabase start`）を使用する
- マイグレーションとシードが適用された状態でテストする
- RLS ポリシーのテストもローカル環境で実施する

### Unit テスト作成方針（要約）

- Unit は純粋ロジック（計算・変換・判定）を優先し、DOM/DB検証はそれぞれ Component / E2E テストへ寄せる。
- `assertTrue(true)` のみで成立するテストは作成しない。
- private 関数の直接テストは避け、public API 経由で検証する。

### 必須ルール

- AAA パターン厳守（Arrange / Act / Assert）
- テストケースの説明は日本語で記載
- モックを使用して依存を分離
- 時間依存のテストは `vi.useFakeTimers()` で固定時刻にする
- E2E テストは各テストが独立して実行可能にする

## ADR（Architecture Decision Record）運用ルール

### 作成タイミング

- **Issue完了時に ADR 作成を推奨する**（設計判断を伴う場合は必須）
- アーキテクチャ上の重要な判断をしたとき
- 新しい技術・ライブラリを導入したとき
- 既存の設計方針を変更・撤回したとき

### 配置場所とファイル名

- 配置先: `docs/design/decisions/`
- ファイル名: `YYYY-MM-DD-{英語kebab-case}.md`（例: `2026-04-06-realtime-chat-architecture.md`）
- テンプレート: `docs/design/decisions/_template.md` を使用する

### 必須セクション

- **背景**: なぜこの判断が必要だったか
- **決定内容**: 何を採用し、どう実装したか
- **代替案**: 検討した他の選択肢とその却下理由
- **結果**: この決定による影響・トレードオフ

### 作成後の手順

- `docs/INDEX.md` の `decisions/` セクションにエントリを追加する

## エージェント運用ルール（詳細）

上記「エージェント運用ルール（必須）」に加え、以下のルールを遵守する。

### ADR 関連

- **Issue完了時に ADR を作成することを推奨する。** 設計判断を含む場合は必須。
- ADR は `docs/design/decisions/_template.md` のフォーマットに従う。
- ADR のステータスは `proposed` → `accepted` → 必要に応じて `deprecated` で管理する。

### ドキュメント作成時の必須事項

- **新規ファイル作成時は必ず Frontmatter（title, description, tags）を付与する。**
- ファイル名は英語 kebab-case、本文は日本語。
- **ドキュメント変更後は `docs/INDEX.md` の更新を忘れずに行う。**

### 実装完了時の確認フロー

1. 関連ドキュメント（画面設計・DB設計・ADR等）の更新要否を確認
2. 関連テストの作成・更新要否を確認
3. `docs/INDEX.md` が最新状態であることを確認
4. 仕様ドキュメントと実装に乖離がないことを確認（SSOT原則）
